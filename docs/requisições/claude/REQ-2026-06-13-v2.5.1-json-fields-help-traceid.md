# REQ — v2.5.1 Bugfixes --json e help

## Contexto

Validação de campo da v2.5.0 no CMDB identificou dois bugs complementares.
Achados em `docs/analise-cmdb/achados-v2.5.0-json-fields-e-docs.md`.

---

## Bug 1 — `--json` não popula `rule` nem `file` (médio)

`trackfw validate --json` emite JSON estruturalmente válido, mas cada item tem
`rule: ""` e `file: ""` — apenas `message` é preenchido. Causa: o validator retorna
`[]string`; o nome da regra e o path do arquivo existem apenas embutidos no texto.

### Correção

Propagar `rule` e `file` desde a origem:
- Em `applyRule(ruleName, msgs, ...)`: construir `RuleItem{Rule: ruleName, File: extractFile(msg), Message: msg}` onde `extractFile` extrai o primeiro path/filename entre aspas da mensagem (heurística que cobre os formatos existentes).
- Mudar o pipeline interno para operar em `[]RuleItem` em vez de `[]string` onde necessário para carregar o ruleName até a serialização.
- Manter `message` inalterado (compatível com saída humana).

---

## Bug 2 — `trace_id_field` e `rules.traceid_*` ausentes do `trackfw help` (menor)

`trackfw help` não lista `trace_id_field` nem as 5 severidades configuráveis
`rules.traceid_duplicate_req`, `rules.traceid_duplicate_roadmap`,
`rules.traceid_orphan_req`, `rules.traceid_orphan_roadmap`, `rules.traceid_state_mismatch`.

### Correção

Adicionar as entradas ao `configDocs` (ou equivalente) de cada CLI. O wizard
`configure` (se listar regras) também deve incluí-las.

---

## Critérios de aceite

- [ ] `trackfw validate --json` → cada item tem `rule` preenchido (nome da regra)
- [ ] `trackfw validate --json` → cada item tem `file` preenchido (path do artefato)
- [ ] `trackfw help` lista `trace_id_field` com default e descrição
- [ ] `trackfw help` lista as 5 `rules.traceid_*` com default e descrição
- [ ] Saída texto do validate inalterada (sem `--json`)
- [ ] Testes atualizados nos 3 CLIs
- [ ] Paridade nos 3 CLIs (Go · Node.js · Python)
