# REQ — v2.4.1 Bugfixes baseline e parser YAML

## Contexto

Validação de campo da v2.4.0 no projeto CMDB (Zeus/CMDB) identificou dois bugs que
comprometem os casos de uso centrais da v2.4.0.
Achados registrados em `docs/analise-cmdb/achados-v2.4.0-baseline-warnings-e-config.md`.

---

## Problema

### Bug 1 — Ratchet do baseline ignora `warnings` (alto impacto)

`Validate()` aplica o set-difference do baseline somente em `violations`, nunca em `warnings`.
Como a maioria do passivo legado vive em `warning` (`adr_orphan`, `stale_wip`, `folder_status`,
`ref_targets_exist` têm default `"warning"`), o baseline é efetivamente inútil para o principal
caso de uso: congelar ruído legado em projetos com histórico.

Em modo `lenient`, o problema é agravado: violations remanescentes são movidas para warnings
**depois** do filtro — logo, warnings baselined passam pelo filtro de violations sem efeito,
e depois são recriados pelo lenient.

### Bug 2 — Valores de regra entre aspas não são reconhecidos (médio)

```yaml
rules:
  adr_orphan: "off"   # COM aspas → NÃO reconhecido; regra continua ativa
  adr_orphan: off     # SEM aspas → funciona
```

O parser de `trackfw.yaml` não remove aspas envolventes do valor antes de comparar com
`off|warning|error`. O agravante: `trackfw help` exibe os valores **entre aspas** (`"error"`,
`"warning"`, `"off"`), induzindo o usuário a copiar exatamente o formato que quebra o parser.

---

## Solução

### Fix 1 — Aplicar ratchet simetricamente a `warnings`

Em `Validate()` de cada CLI, após filtrar violations com baseline, filtrar warnings
pelo mesmo mecanismo:

```go
warnSet := make(map[string]struct{}, len(baseline.Warnings))
for _, w := range baseline.Warnings {
    warnSet[w] = struct{}{}
}
var netNewWarn []string
for _, w := range warnings {
    if _, exists := warnSet[w]; !exists {
        netNewWarn = append(netNewWarn, w)
    }
}
warnings = netNewWarn
```

**Ordem obrigatória**: filtrar baseline (violations E warnings) **antes** de aplicar lenient mode,
para que o lenient não recrie warnings já baselined.

### Fix 2 — Trim de aspas no parser YAML

Em cada CLI, no ponto onde valores de sub-chaves indentadas são extraídos (ex: regras em `rules:`),
aplicar trim de aspas simples e duplas após o trim de espaços:

- Go: em `parse()` em `internal/config/config.go`, linha que extrai `val` de sub-chave indentada
- Node.js: em `_parse()` em `npm/src/config/index.js`
- Python: em `_parse()` em `pypi/trackfw/config.py`

Já existe precedente: `extractFrontmatterField` em todos os CLIs já faz este trim — apenas tornar consistente no parser de config.

## Critérios de aceite

- [ ] `trackfw baseline` com repo contendo warnings → `.trackfw-baseline.json` os persiste (já funciona)
- [ ] `trackfw validate` após baseline → warnings baselined **não** aparecem no output
- [ ] Em modo `lenient`, warnings baselined também são suprimidos
- [ ] `rules:\n  adr_orphan: "off"` (com aspas) → regra desativada (igual a sem aspas)
- [ ] `rules:\n  stale_wip: "warning"` (com aspas) → severidade warning aplicada
- [ ] Testes novos cobrindo ambos os cenários nos 3 CLIs
- [ ] Paridade nos 3 CLIs (Go · Node.js · Python)
