# REQ — v2.4 Config evolution

## Contexto

A v2.3.0 rodada no projeto CMDB (monorepo legado real) produziu 73 avisos em modo `lenient`, dos quais
a maioria não é dívida real:

- 7 falsos positivos de "wip sem REQ" — o CMDB usa `req_id:` no frontmatter, não a linha `REQ:` que o
  validador casa por substring hardcoded.
- 59 falsos positivos de "ADR órfão" — ADRs que precedem a adoção do fluxo REQ→ADR (passivo histórico).

Análise completa: `docs/analise-cmdb/evolucao-generica-baseline-legado.md`

---

## Problema

O trackfw atual tem dois pontos de fragilidade para adoção em projetos reais:

1. **Convenção hardcoded** — marcadores de link (`REQ:`, `ADR:`, `Roadmap:`) e critérios de aceite
   (`## Acceptance Criteria`, `## Critérios de Aceite`) são strings fixas no código. Projetos com
   convenções próprias geram falsos positivos sem saída.

2. **Legado sem escada de saída** — `lenient`/`lenient_until` é um interruptor global que esconde tudo.
   Não distingue passivo histórico (aceitável, congelar) de trabalho novo negligente (deve bloquear).
   O resultado é: ou o time aguenta o ruído e abandona o gate, ou desliga tudo e perde a governança.

3. **Rigor liga/desliga** — não há como ligar regras progressivamente; times em maturação precisam de
   uma escada (off → warning → error) por regra, não de um toggle global.

4. **Opacidade de configuração** — o `trackfw.yaml` exige conhecer os campos de cabeça; não há help
   integrado nem wizard para gerar a configuração mínima necessária.

---

## Solução

### F1 — Config sparse com defaults explícitos (fundação)

Todos os campos do `trackfw.yaml` passam a ter defaults documentados e hardcoded nos 3 CLIs.
O arquivo `trackfw.yaml` de um projeto só precisa conter o que diverge do padrão.
Projetos dentro da convenção default funcionam com arquivo mínimo (ou sem arquivo).

Novos campos a suportar no parser dos 3 CLIs (seções aninhadas de 1 nível):

```yaml
link_fields:
  req:
    - "REQ:"
  adr:
    - "ADR:"
  roadmap:
    - "Roadmap:"

acceptance_markers:
  - "## Acceptance Criteria"
  - "## Critérios de Aceite"

rules:
  wip_has_req:         error
  wip_acceptance:      error
  wip_limit:           error
  stale_wip:           warning
  adr_orphan:          warning
  ref_targets_exist:   warning
  folder_status:       warning
  filename_uniqueness: error
  blocked_by_draft_adr: error
```

### F2 — Field mapping (R1 da análise CMDB)

Validadores passam a consultar `link_fields` da config em vez de strings hardcoded.
Um documento que declara `req_id: REQ-001` no frontmatter satisfaz "tem REQ" se `link_fields.req`
incluir `req_id`.

Marcadores de critério de aceite passam a usar a lista `acceptance_markers` da config.

### F3 — Severidade por regra (R3)

Cada regra do validador passa a ter severidade configurável: `off | warning | error`.
- `error` → conta como violation (bloqueia gate, status ≠ 0)
- `warning` → conta como warning (exibido mas não bloqueia)
- `off` → regra desativada silenciosamente
- Defaults documentados: regras críticas de processo em `error`, regras de qualidade em `warning`

### F4 — Baseline snapshot + catraca (R2 da análise CMDB)

Novo comando `trackfw baseline`:
- Executa `validate` internamente e grava o resultado em `.trackfw-baseline.json`
- Formato: `{"created": "ISO8601", "violations": [...], "warnings": [...]}`

Comportamento de `trackfw validate` com baseline presente:
- Compara violations atuais com as do baseline
- Falha **somente** em violations novas (não presentes no baseline)
- Warnings novos são exibidos mas não bloqueiam
- Catraca: baseline nunca é atualizado automaticamente — só via `trackfw baseline` explícito

`.trackfw-baseline.json` deve ser commitado no repositório (é documentação do passivo aceito).

### F5 — `trackfw help <key>` (man-page integrado)

Novo subcomando `trackfw help <key>`:
- Sem argumento: lista todas as keys configuráveis com descrição de uma linha
- Com argumento: exibe descrição completa, tipo, default, exemplo e impacto no gate

### F6 — Wizard `trackfw configure`

Novo comando `trackfw configure`:
- Detecta `trackfw.yaml` existente e oferece atualizar ou recriar
- Faz perguntas sobre convenção do projeto (link markers, dirs, wip limit, etc.)
- Gera `trackfw.yaml` com **apenas** os campos que divergem dos defaults
- Integra com `trackfw help` para exibir contexto em cada pergunta

---

## Critérios de aceite globais

- [ ] `trackfw.yaml` sem `link_fields`/`rules`/`acceptance_markers` → defaults se aplicam, comportamento idêntico à v2.3
- [ ] Projeto com `req_id:` no frontmatter + `link_fields.req: [req_id]` → sem falsos positivos de "wip sem REQ"
- [ ] `trackfw baseline` grava `.trackfw-baseline.json` com violations atuais
- [ ] `trackfw validate` com baseline → falha somente em violations novas
- [ ] Regra com `off` → não aparece no output
- [ ] Regra com `warning` → aparece em warnings, não bloqueia (exit 0)
- [ ] Regra com `error` → bloqueia (exit 1)
- [ ] `trackfw help` sem arg → lista todas as keys
- [ ] `trackfw help wip_limit` → exibe doc completa da key
- [ ] `trackfw configure` → gera `trackfw.yaml` mínimo (somente não-defaults)
- [ ] Paridade nos 3 CLIs para F1–F5; wizard (F6) pode variar na UX mas produz o mesmo YAML

---

## Não está no escopo

- Parser YAML completo (continua linha a linha, sem biblioteca externa)
- Suporte a mais de 1 nível de aninhamento além dos blocos definidos
- UI web ou TUI avançada para o wizard
- `trackfw baseline diff` (comparação visual — v2.5)
