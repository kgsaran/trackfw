# Roadmap v2.0 — Implementação dos Gaps trackfw

> Criado em: 2026-06-13 | Concluído: 2026-06-13 | Status: ✅ Done
> REQ: [REQ-2026-06-13-gaps-v2-implementacao](../../../requisições/claude/REQ-2026-06-13-gaps-v2-implementacao.md)

## Diagnóstico / Contexto

Análise comparativa do trackfw vs. mercado (adr-tools, log4brains, Backstage, Linear, Cortex.io)
identificou 7 gaps distribuídos em 3 prioridades. Este roadmap implementa todos eles em ordem
decrescente de impacto, maximizando paralelismo entre MLs independentes.

**Branch a criar antes de iniciar:** `feat/v2.0-gaps`

**Status:** ⬜ Pendente
**Agente:** Apolo
**Branch:** `feat/v2.0-gaps`
**Arquivos afetados:**
- `internal/commands/serve.go` (novo)
- `internal/server/server.go` (novo)
- `internal/server/templates/index.html` (novo)
- `internal/server/templates/adr.html` (novo)
- `internal/commands/root.go` (registrar serve)

**Ações:**
1. Criar `internal/server/server.go`:
   - `Start(port int)` sobe `net/http` na porta (default 4080)
   - Handler `/` renderiza página com 3 seções: grafo, timeline, kanban

3. Registrar `newServeCmd()` em `root.go`
4. Templates HTML com CSS inline (sem dependência de CDN): grid layout, cores do lipgloss palette

**Critérios de aceite:**
- [ ] `trackfw serve` sobe sem erro em projeto com docs/adr/, docs/req/, docs/roadmaps/

- [ ] Timeline mostra ADRs ordenados por data do filename

   - Mesma lógica de parse do `.trackfw-log` em JS (sem deps externas)
   - Flag `--since` e `--export` idênticas
   - Saída em texto puro (sem lipgloss)

**Critérios de aceite:**
- [ ] `trackfw metrics` com log vazio: imprime "No transitions recorded yet."
- [ ] `trackfw metrics` com log preenchido: exibe cycle time, throughput, WIP age
- [ ] `trackfw metrics --since 30d` filtra corretamente
- [ ] `trackfw metrics --export metrics.csv` gera arquivo CSV com headers

### ML-2A — Brownfield Onboarding (`trackfw init --brownfield`)
**Status:** ⬜ Pendente
**Agente:** Apolo
**Branch:** `feat/v2.0-gaps`
**Arquivos afetados:**
- `internal/commands/init.go` (flag `--brownfield`)
- `internal/generators/scaffold.go` (campo `BrownfieldMode`, `LenientUntil`)
- `internal/validator/validator.go` (ler `governance_mode` do `trackfw.yaml`)
- `npm/src/generators/init.js` (paridade)
- `npm/src/validator/index.js` (paridade)

**Ações:**
1. Adicionar flag `--brownfield` em `internal/commands/init.go`:
   - Se `--brownfield`: `cfg.BrownfieldMode = true`, `cfg.LenientUntil = time.Now().AddDate(0, 0, 30)`
2. Em `internal/generators/scaffold.go`:
   - `writeTrackfwConfig` adiciona ao YAML: `governance_mode: lenient` e `lenient_until: YYYY-MM-DD` se `BrownfieldMode`
3. Em `internal/validator/validator.go`:
   - `readGovernanceMode()` lê `trackfw.yaml` e retorna `strict` (default) ou `lenient` + `lenient_until`
   - Se modo `lenient` e `time.Now().Before(lenientUntil)`: violations → `[WARN]`, exit code 0
   - Se `lenient_until` já passou: modo vira `strict` automaticamente (log mensagem explicativa)
4. Paridade npm em `npm/src/generators/init.js` e `npm/src/validator/index.js`

**Critérios de aceite:**
- [ ] `trackfw init --brownfield` gera `trackfw.yaml` com `governance_mode: lenient` e `lenient_until` 30 dias à frente
- [ ] `trackfw validate` em projeto lenient com violações: saída com `[WARN]`, exit code 0
- [ ] `trackfw validate` em projeto lenient com `lenient_until` no passado: volta ao modo strict
- [ ] `go build ./...` e testes verdes

**Comandos de validação:**
```bash
go build ./...
go test ./internal/validator/...
go test ./internal/generators/...
```

   - `LinearClient{APIKey, TeamID string}`
   - `CreateIssue(title, description string) (issueID string, err error)` — POST `https://api.linear.app/graphql`
   - GraphQL mutation: `issueCreate(input: {title, description, teamId})`

2. Criar `internal/sync/jira.go`:
   - `JiraClient{BaseURL, Email, Token, Project string}`

   - Flag `--to=linear` ou `--to=jira`
   - Lê todos os REQs em `docs/req/*.md` com `Status: Open` e sem `linear_issue`/`jira_issue`
   - Para cada REQ: chama CreateIssue, atualiza frontmatter do arquivo com o ID retornado
   - Imprime tabela de REQs sincronizadas
4. Em `internal/generators/req.go`:
   - Template de REQ inclui linhas `| Linear Issue: ` e `| Jira Issue: ` no frontmatter
5. Paridade npm: `npm/src/commands/sync.js` (stdlib `https` apenas)

**Critérios de aceite:**
- [ ] `trackfw sync --to=linear` com `LINEAR_API_KEY` e `LINEAR_TEAM_ID` válidos: cria issue e atualiza REQ
- [ ] `trackfw sync --to=jira` com config válida: cria issue e atualiza REQ
