# Roadmap v2.5.1 — Bugfix --json rule/file + help trace_id_field

> Criado em: 2026-06-13 | Status: ✅ Done

## Diagnóstico

Dois bugs complementares identificados na validação de campo da v2.5.0 no CMDB:

1. `validate --json` emite `rule: ""` e `file: ""` em cada item — `applyRule` conhece o
   `ruleName` mas não o propaga ao `RuleItem`; `extractFile` não existe.
2. `trackfw help` não lista `trace_id_field` nem as 5 `rules.traceid_*` adicionadas na v2.5.0.

Ambos são ortogonais entre si e entre CLIs → 3 MLs em paralelo (um por CLI, cada um resolve os 2 bugs).

---

## Wave 1 — Fix --json rule/file + help traceid (3 MLs em paralelo)

> Dependências: independente

### ML-1A — Go

**Status:** ✅ Concluído  
**Arquivos afetados:**
- `internal/validator/result.go` — `TaggedMsg` struct, `extractFile`, `BuildResultTagged`
- `internal/validator/validator.go` — `applyRuleTagged`, `validateUnfilteredTagged`, `ValidateTagged`
- `internal/commands/validate.go` — usa `ValidateTagged` + `BuildResultTagged` no modo `--json`
- `internal/commands/help.go` — 6 entradas adicionadas ao `configDocs`
- `internal/commands/validate_json_test.go` — verifica `rule`/`file` preenchidos

**Abordagem adotada:** pipeline tagged paralelo ao existente (`ValidateTagged` retorna `[]TaggedMsg`).
`applyRuleTagged` injeta o `ruleName` nos `TaggedMsg`; `BuildResultTagged` converte em `RuleItem`
via `extractFile` (regex `"([^"]+)"`). API pública `Validate()` inalterada (sem regressões).

**Critérios de aceite:**
- [x] `trackfw validate --json` → itens com `rule` e `file` preenchidos
- [x] `trackfw help` lista `trace_id_field` e as 5 `rules.traceid_*`
- [x] `go test ./internal/... -q` verde sem regressões
