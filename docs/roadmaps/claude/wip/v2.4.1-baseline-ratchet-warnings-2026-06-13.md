## Wave 1 — Fix B1: ratchet simétrico de warnings no baseline (3 MLs em paralelo)

> MLs independentes — cada CLI em arquivo distinto.

### ML-1A — Go
**Status:** ⬜ Pendente

**Arquivo afetado:** `internal/validator/validator.go`

**Mudança em `Validate()`:**

Localizar o bloco do ratchet de violations (após `ValidateUnfiltered()`, antes do lenient mode).
Logo após filtrar violations, aplicar o mesmo set-difference em warnings:

```go
// Após: violations = netNew
warnSet := make(map[string]struct{}, len(baseline.Warnings))
for _, w := range baseline.Warnings {
    warnSet[w] = struct{}{}
}
var netNewWarn []string
for _, w := range warnings {
    if _, exists := warnSet[w]; !exists {
        netNewWarn = append(netNewWarn, w)
    }
}
warnings = netNewWarn
// Só então aplicar lenient mode
```

**Ordem obrigatória:** filtro de baseline (violations + warnings) → lenient mode. Nunca inverter.

**Testes a adicionar** em `internal/validator/validator_baseline_test.go`:
- `TestBaselineFiltersWarnings`: repo com warning (ex: ADR órfão com severidade "warning") → após baseline, warning não aparece em `Validate()`
- `TestBaselineLenientNoRecreate`: modo lenient + baseline → warning baselined não reaparece (lenient não recria)

**Critérios de aceite:**
- [ ] `go test ./internal/validator/...` verde com 2 novos testes

### ML-1B — Node.js
**Status:** ⬜ Pendente

**Arquivo afetado:** `npm/src/validator/index.js`

**Mudança em `validate()`:**

Localizar o bloco do ratchet (após `validateUnfiltered()`, antes do lenient mode).
Logo após filtrar violations, aplicar set-difference em warnings:

```js
// Após: violations = violations.filter(v => !baselineSet.has(v))
const baselineWarnSet = new Set(baseline.warnings || [])
warnings = warnings.filter(w => !baselineWarnSet.has(w))
```

## Wave 2 — Fix B2: trim de aspas no parser YAML (3 MLs em paralelo)

> Dependência: Wave 1 completa (evitar conflitos em validator.go).

### ML-2A — Go
**Status:** ⬜ Pendente

**Arquivo afetado:** `internal/config/config.go`

**Mudança:**

Na função `parse()`, localizar o trecho que processa sub-chaves indentadas do bloco `rules:`:

```go
// Trecho atual (aproximado):
k = strings.TrimSpace(...)
v = strings.TrimSpace(line[colon_idx+1:])
rules[k] = v
```

Adicionar trim de aspas no valor **antes** de armazenar:

```go
v = strings.Trim(v, "\"'")
rules[k] = v
```

Aplicar o mesmo trim a **todos** os valores escalares extraídos pelo parser (campos top-level e sub-chaves), para consistência e prevenção de regressão futura. Verificar se `extractFrontmatterField` já faz isso (sim, já faz) e tornar o parser de config igualmente robusto.

**Testes a adicionar** em `internal/config/config_evolution_test.go`:
- `TestRulesValueWithDoubleQuotes`: `rules:\n  adr_orphan: "off"` → `cfg.Rules["adr_orphan"] == "off"`
- `TestRulesValueWithSingleQuotes`: `rules:\n  stale_wip: 'warning'` → `cfg.Rules["stale_wip"] == "warning"`

**Critérios de aceite:**
- [ ] `go build ./...` limpo
- [ ] `go test ./internal/config/...` verde com 2 novos testes

### ML-2B — Node.js
**Status:** ⬜ Pendente

**Arquivo afetado:** `npm/src/config/index.js`

**Mudança:**

Na função `_parse()`, localizar o trecho que processa sub-chaves indentadas do bloco `rules:`:

```js
// Trecho atual (aproximado):
const v = line.slice(colonIdx + 1).trim()
rules[k] = v
```

Adicionar trim de aspas:

```js
const v = line.slice(colonIdx + 1).trim().replace(/^["']|["']$/g, '')
rules[k] = v
```

Aplicar o mesmo replace a todos os valores escalares extraídos pelo parser para consistência.

**Testes a adicionar** em `npm/tests/config.test.js`:
- `rules com aspas duplas são reconhecidas`: `rules:\n  adr_orphan: "off"` → `cfg.rules.adr_orphan === "off"`
- `rules com aspas simples são reconhecidas`: `rules:\n  stale_wip: 'warning'` → `cfg.rules.stale_wip === "warning"`

**Critérios de aceite:**
