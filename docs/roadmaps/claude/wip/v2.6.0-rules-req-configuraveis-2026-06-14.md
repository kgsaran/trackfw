# Roadmap v2.6.0 — Rules REQ configuráveis

> Criado em: 2026-06-14 | Status: 🔄 WIP

## Diagnóstico

`validateREQsHaveADR`, `validateREQsHaveRoadmap` e `validateBlockedHasREQ` são sempre-erro
nos 3 CLIs. Todas as outras regras já passam por `applyRule`/`_apply_rule` e suportam
`off`/`warning`/`error`. A assimetria impede adotantes de afrouxar essas 3 regras sem
forkar o código.

**Callsites afetados (substituição direta):**

| CLI | Arquivo | Linhas | Troca |
|-----|---------|--------|-------|
| Go | `internal/validator/validator.go` | ~310, ~316, ~322 | `violations = append(...)` → `applyRule(...)` |
| Go | `internal/validator/validator.go` | ~445–463 | loop `TaggedMsg{Rule:""}` → `applyRuleTagged(...)` |
| Node.js | `npm/src/validator/index.js` | ~823–825 | loop `violations.push` → `applyRule(...)` |
| Python | `pypi/trackfw/validator.py` | ~892–894 | `violations += _enrich_items(...)` → `_apply_rule(...)` |

### ML-1A — Go

**Status:** ⬜ Pendente

**Arquivos afetados:**
- `internal/validator/validator.go`
- `internal/validator/validator_test.go`

**Ações:**

1. Em `validateUnfiltered` (~linha 306–322), substituir:
   ```go
   violations = append(violations, reqViolations...)       // sem regra configurável
   violations = append(violations, blockedViolations...)   // sem regra configurável
   violations = append(violations, reqRoadmapViolations...) // sem regra configurável
   ```
   Por:
   ```go
   applyRule("req_has_adr",     reqViolations,        &violations, &warnings)
   applyRule("blocked_has_req", blockedViolations,    &violations, &warnings)
   applyRule("req_has_roadmap", reqRoadmapViolations, &violations, &warnings)
   ```

2. Em `validateUnfilteredTagged` (~linha 441–463), substituir os 3 loops:
   ```go
   for _, m := range reqViolations {
       violations = append(violations, TaggedMsg{Rule: "", Msg: m})
   }
   // idem para blockedViolations e reqRoadmapViolations
   ```
   Por:
   ```go
   applyRuleTagged("req_has_adr",     reqViolations,        &violations, &warnings)
   applyRuleTagged("blocked_has_req", blockedViolations,    &violations, &warnings)
   applyRuleTagged("req_has_roadmap", reqRoadmapViolations, &violations, &warnings)
   ```

3. Adicionar/atualizar testes em `validator_test.go`:
   - `TestReqHasADRConfiguravel`: config `rules.req_has_adr: "warning"` → violation vira warning; `"off"` → silenciado; sem config → violation (default)
   - `TestBlockedHasREQConfiguravel`: idem para `blocked_has_req`
   - `TestReqHasRoadmapConfiguravel`: idem para `req_has_roadmap`
   - Verificar que em `--json` (via `BuildResultTagged`), os itens carregam `"rule": "req_has_adr"` etc.

### ML-1B — Node.js

**Status:** ⬜ Pendente

**Arquivos afetados:**

- `npm/tests/rules_req_configuraveis.test.js` (novo)

**Ações:**

1. Em `validate` (~linhas 823–825), substituir:
   ```js
   for (const msg of validateREQsHaveADR())    { _setMeta(msg, 'req_has_adr');    violations.push(msg) }
   for (const msg of validateBlockedHasREQ())   { _setMeta(msg, 'blocked_has_req'); violations.push(msg) }
   for (const msg of validateREQsHaveRoadmap()) { _setMeta(msg, 'req_has_roadmap'); violations.push(msg) }
   ```
   Por:
   ```js
   applyRule('req_has_adr',     validateREQsHaveADR(),     violations, warnings)
   applyRule('blocked_has_req', validateBlockedHasREQ(),   violations, warnings)
   applyRule('req_has_roadmap', validateREQsHaveRoadmap(), violations, warnings)
   ```
   Nota: `applyRule` já chama `_setMeta` internamente para as regras que passa — verificar se isso se aplica ou se é preciso ajustar `applyRule` para popular `_itemMeta` também para essas regras.

   - `rules.req_has_adr: "off"` → msg silenciada
   - Sem config → msg em violations (default error)
   - Idem para `blocked_has_req` e `req_has_roadmap`
