# Roadmap v2.4 — Config evolution

> Criado em: 2026-06-13 | Status: 🔄 WIP
> REQ: [REQ-2026-06-13-v2.4-config-evolution](../../../requisições/claude/REQ-2026-06-13-v2.4-config-evolution.md)
> Branch: `feat/v2.4-config-evolution`

**REGRA DURA:** Paridade nos 3 CLIs (Go + Node.js + Python) em todas as waves. Exceção explícita: UX do wizard (F6) pode variar, mas o YAML gerado deve ser idêntico.

**Dependência de waves:** Wave 1 → Wave 2 → Wave 3 → Wave 4. Dentro de cada wave, os 3 MLs são paralelos.

---

## Diagnóstico

| # | Problema | Impacto |
|---|----------|---------|
| P1 | Marcadores de link hardcoded (`REQ:`, `ADR:`) | Falsos positivos em projetos com convenção própria |
| P2 | `lenient` global esconde todo o passivo sem distinção | Adoção zero ou gate inútil |
| P3 | Rigor liga/desliga global | Times em maturação não conseguem adoção progressiva |
| P4 | Configuração opaca, sem help integrado | Onboarding difícil, wizard ausente |

```go

    return ProjectConfig{
        ADRDirs:            []string{"docs/adr"},
        REQDir:             "docs/req",
        RoadmapDir:         "docs/roadmaps",
        RoadmapNamespacing: "flat",
        WipLimit:           1,
        LinkFieldsReq:      []string{"REQ:"},
        LinkFieldsADR:      []string{"ADR:"},
        LinkFieldsRoadmap:  []string{"Roadmap:"},
        AcceptanceMarkers:  []string{"## Acceptance Criteria", "## Critérios de Aceite"},
        Rules: map[string]string{
            "wip_has_req":          "error",
            "wip_acceptance":       "error",
            "wip_limit":            "error",
            "stale_wip":            "warning",
            "adr_orphan":           "warning",
            "ref_targets_exist":    "warning",
            "folder_status":        "warning",
            "filename_uniqueness":  "error",
            "blocked_by_draft_adr": "error",
        },
    }
}
```

Estender `parse()` para suportar blocos aninhados de 1 nível. Adicionar estados:
- `inLinkFieldsReq`, `inLinkFieldsADR`, `inLinkFieldsRoadmap` (listas após `  req:`, `  adr:`, `  roadmap:`)
- `inAcceptanceMarkers` (lista após `acceptance_markers:`)
- `inRules` (pares `  key: value` após `rules:`)
- `inLinkFields` (bloco após `link_fields:`)

Lógica do parser estendido (pseudocódigo):
```
ao encontrar "link_fields:" → inLinkFields = true
ao encontrar "  req:" dentro de inLinkFields → inLinkFieldsReq = true
ao encontrar "  - valor" dentro de inLinkFieldsReq → append(linkFieldsReq, valor)
ao encontrar "  adr:" dentro de inLinkFields → flush req, inLinkFieldsADR = true
ao encontrar "  roadmap:" dentro de inLinkFields → flush adr, inLinkFieldsRoadmap = true
ao encontrar "acceptance_markers:" → inAcceptanceMarkers = true
ao encontrar "  - valor" dentro de inAcceptanceMarkers → append(acceptanceMarkers, valor)
ao encontrar "rules:" → inRules = true
ao encontrar "  key: value" dentro de inRules → rules[key] = value
qualquer linha sem prefixo de espaço → flush estado atual e sair do bloco aninhado
```

**Testes a adicionar em `internal/config/`:**
- `TestConfigDefaults`: sem `trackfw.yaml` → todos os defaults esperados
- `TestConfigLinkFields`: yaml com `link_fields.req: [req_id, "REQ:"]` → `LinkFieldsReq = ["req_id", "REQ:"]`
- `TestConfigAcceptanceMarkers`: yaml com lista customizada → aplicado
- `TestConfigRules`: yaml com `rules:\n  stale_wip: error` → `Rules["stale_wip"] == "error"`, outros mantêm default
- `TestConfigSparse`: yaml somente com `wip_limit: 3` → restante usa defaults

**Critérios de aceite:**
- [ ] `go build ./...` sem erros
- [ ] `go test ./internal/config/...` verde

**Mudança F3 — Severidade por regra:**

Adicionar helper:
```go
func ruleSeverity(name string) string {
    cfg := config.Load()
    if s, ok := cfg.Rules[name]; ok { return s }
    return "error" // fallback seguro
}
```

Adicionar helper para emitir resultado conforme severidade:
```go
func applyRule(name, message string, violations, warnings *[]string) {
    switch ruleSeverity(name) {
    case "off":     // silencioso
    case "warning": *warnings = append(*warnings, message)
    default:        *violations = append(*violations, message)
    }
}
```

Substituir todos os `violations = append(violations, msg)` e `warnings = append(warnings, msg)` por `applyRule("nome_da_regra", msg, &violations, &warnings)` nos seguintes pontos:
- `wip_has_req` → `validateWIPHasREQ`
- `wip_acceptance` → `validateWIPHasAcceptanceCriteria`
- `wip_limit` → `validateWIPLimit`
- `stale_wip` → `validateStaleWIP`
- `adr_orphan` → `validateADRsAreReferenced`
- `ref_targets_exist` → `validateRefTargetsExist`
- `folder_status` → `validateFolderStatusCoherence`
- `filename_uniqueness` → `validateFilenameUniqueness`
- `blocked_by_draft_adr` → `validateREQsNotBlockedByDraftADRs`

### ML-4A — Go
**Status:** ⬜ Pendente

**Arquivos afetados:**
- `internal/commands/help.go` (novo) — `trackfw help [key]`
- `internal/commands/configure.go` (novo) — `trackfw configure` (wizard com `huh`)

**`trackfw help [key]`:**

Definir mapa interno `configDocs`:
```go
var configDocs = map[string]ConfigKeyDoc{
    "adr_dirs": {
        Type:        "list of strings",
        Default:     `["docs/adr"]`,
        Description: "Diretórios onde os ADRs são armazenados. Suporta múltiplos caminhos.",
        Example:     "adr_dirs:\n  - docs/adr\n  - docs/architecture/decisions",
        Impact:      "Afeta todas as verificações de ADR (órfão, draft, referência).",
    },
    "wip_limit": { ... },
    "link_fields.req": { ... },
    "rules.stale_wip": { ... },
    // ... todos os campos
}
```
- Sem argumento: lista todas as keys em formato tabular (key | default | descrição curta)
- Com argumento: exibe doc completa da key

**`trackfw configure` (wizard):**

Usar `huh` para perguntas interativas. Fluxo:
1. Detectar `trackfw.yaml` existente → oferecer atualizar ou recriar
2. Perguntar: `adr_dirs` (default: `docs/adr`)
3. Perguntar: `req_dir`, `roadmap_dir`, `wip_limit`
4. Perguntar: convenção de link REQ (`REQ:` inline ou campo de frontmatter)
5. Perguntar: severidade de regras (exibir lista, permitir ajustar individualmente)
6. Gerar `trackfw.yaml` com **somente** os campos que diferem dos defaults

**Critérios de aceite:**
- [ ] `trackfw help` lista todas as keys
- [ ] `trackfw help wip_limit` exibe doc completa
- [ ] `trackfw configure` gera `trackfw.yaml` mínimo (sem campos no valor default)
- [ ] `go build ./...` + `go test ./...` verdes
