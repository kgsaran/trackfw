# Roadmap: v2.7.0 — trackfw serve Dashboard Web

> Criado em: 2026-06-14 | Status: 🔄 WIP

## Diagnóstico / Contexto

O `trackfw serve` hoje é um stub HTTP sem UI. Os dados necessários para um
dashboard rico já existem: artefatos MD em disco, grafo de dependências via
`validate --json`, log de transições em `.trackfw-log`. A feature é exclusiva
Go (`trackfw serve` não existe nos CLIs Node.js/Python — exceção de paridade).

Tecnologia: HTMX + marked.js + Chart.js servidos via `embed.FS` — single-binary,
zero dependência de runtime externo.

---

### ML-1D — Endpoint `GET /api/metrics`
**Status:** ⬜ Pendente
**Arquivos afetados:**
- `internal/serve/api_metrics.go` (novo)
- `internal/serve/metrics_log.go` (novo — parser do `.trackfw-log`)

**Ações:**
1. Criar parser de `.trackfw-log`:
   - Formato: `YYYY-MM-DD HH:MM  <filename>  <fromState> → <toState>`
   - Construir mapa de transições por arquivo
2. Calcular:
   - `lead_time_avg_days`: data primeira entrada em backlog → data done (média)
   - `cycle_time_avg_days`: data primeira entrada em wip → data done (média)
   - `abandonment_rate`: % de roadmaps em abandoned
   - `state_distribution`: `{ "wip": N, "backlog": N, "done": N, ... }`
   - `burndown`: array `[{ "date": "YYYY-MM-DD", "open": N, "closed": N }]` por semana
3. Retornar JSON com todos os campos

**Critérios de aceite:**
- [ ] `curl localhost:8080/api/metrics` retorna JSON com todos os campos
- [ ] Se `.trackfw-log` não existir, retorna métricas zeradas sem erro
- [ ] Burndown tem pelo menos 1 ponto por semana com dados reais
